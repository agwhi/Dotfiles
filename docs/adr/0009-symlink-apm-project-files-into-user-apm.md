# Symlink APM Project Files Into User APM

Status: accepted.
Date: 2026-07-03.
Amended: 2026-07-07. Reconstructed decision history and ADR quality structure.
Implements: ADR-0008.
Related: ADR-0001, ADR-0002, ADR-0003, ADR-0008,
system/ai/apm/README.md, system/ai/apm/apm.yml,
system/ai/apm/apm.lock.yaml, scripts/setup_symlinks.sh, scripts/doctor.py.

## Context

ADR-0008 selects APM as the AI Asset Manager. APM global mode expects the user
APM project at `~/.apm`, while this repo must remain the source of truth for
the Global AI Baseline manifest and lockfile.

During the AI stabilization work, a repo-owned Codex skill tree was considered
and rejected. The source of truth should be the APM project files, not generated
Codex output or a hand-maintained tool-specific copy.

## Decision Drivers

- The repo should own the APM manifest and lockfile because they are part of
  the new-laptop rebuild and doctor baseline.
- APM should consume those files from its expected user project location.
- Generated target output should remain generated by APM, not committed as the
  primary source of truth.
- Existing `~/.apm` files should be backed up by normal dotfiles symlink setup
  before replacement.
- Future agents should be able to inspect one repo location for package
  evidence and one live symlink location for APM global mode.

## Considered Options

- Make `~/.apm` the source of truth: rejected because it would put durable
  package evidence outside the repo.
- Copy `apm.yml` and `apm.lock.yaml` into `~/.apm`: rejected because copies
  drift unless every update includes a second synchronization step.
- Commit generated Codex, Claude, or opencode target output as source: rejected
  because generated output is target-specific and should be recreated from the
  APM package evidence.
- Symlink repo-owned APM project files into `~/.apm`: selected because it
  matches the existing dotfiles model and keeps one source of truth.

## Decision

The Orchestrator Repo owns the APM project files for the Global AI Baseline:

- `system/ai/apm/apm.yml`
- `system/ai/apm/apm.lock.yaml`

The live user APM project consumes those files through normal dotfiles
symlinks:

- `~/.apm/apm.yml` -> `system/ai/apm/apm.yml`
- `~/.apm/apm.lock.yaml` -> `system/ai/apm/apm.lock.yaml`

Do not keep a repo-owned Codex skill tree as the primary source of truth. Do
not symlink `system/ai/codex/skills/grill-with-docs` into `~/.codex`.

## Consequences

- APM global mode can use the repo-owned manifest and lockfile without making
  `~/.apm` authoritative.
- Running normal symlink setup may back up existing `~/.apm/apm.yml` or
  `~/.apm/apm.lock.yaml` before replacing them with symlinks.
- APM may rewrite deployment metadata in the symlinked lockfile during target
  installs. If that changes canonical package evidence, restore the lockfile
  and verify doctor reports it canonical.
- Target writes remain separate approval gates. Linking the APM project files
  does not automatically approve writes into Codex, Claude Code, opencode, or
  future harness homes.
- The downside is that global APM commands now operate on symlinked repo files,
  so future APM mutation commands need extra care.
